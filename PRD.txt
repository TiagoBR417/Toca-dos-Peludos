📝 PRD: Sistema de Gestão e Adoção – Toca dos Peludos
1. Objetivo e Visão
Facilitar o encontro entre animais resgatados e adotantes responsáveis, centralizando o inventário da ONG e automatizando o fluxo de interesse em adoção.
Problema: Processo de adoção descentralizado e dificuldade em manter o status dos animais atualizado.
Meta: Reduzir o tempo de resposta aos interessados e dar visibilidade aos animais,a ONG e facilitar a inscrição de novos voluntários e denúncias de maus tratos anônimas.
2. Personas (Público-Alvo)
Adotante: Pessoa física que busca um pet e quer filtrar por características (porte, idade, temperamento,raça).
Administrador da ONG: Voluntário que cadastra animais, atualiza fotos e gerencia os pedidos de adoção.

4. Requisitos Não Funcionais (O "Como deve ser")
Performance: A listagem de animais deve carregar em menos de 2 segundos (importante para o SEO e retenção).
Disponibilidade: O sistema deve ser responsivo (funcionar bem em celulares).
Segurança: Dados dos adotantes (LGPD) devem ser armazenados de forma segura na API.
5. Fora de Escopo (O que não faremos agora)
Pagamento de doações via cartão de crédito (será apenas exibido o PIX da ONG).
Chat em tempo real (o contato inicial será via formulário/WhatsApp).
6. Critérios de Aceite
Um animal cadastrado deve aparecer imediatamente na lista pública.
O formulário de interesse não pode ser enviado sem um e-mail válido.
Apenas usuários autenticados (Admin) podem deletar registros de animais.

7. Estrutura do Banco (PostgreSQL via JPA)
O schema é gerenciado automaticamente pelo Hibernate (ddl-auto=update), a partir das entidades:
- tb_pets: id, nome, tipo, raca, porte (ENUM texto), cor, imagemUrl, descricao, status (ENUM texto)
- doacao: id, nomeDoador, emailDoador, valor, dataDoacao
- tb_voluntarios: id, nome, email, telefone, motivos, competencias
- tb_denuncias: id, descricao, localizacao, contato, anonimo
- tb_eventos: id, titulo, data, local, descricao, parceiro
- tb_usuarios_db: id, nome, email (único), senha, endereco, preferencias, tipo (ENUM texto)

 10. Requisitos de Segurança e Privacidade
Para garantir a integridade dos dados da ONG e a privacidade dos usuários, o sistema seguirá os seguintes protocolos:
10.1. Controle de Acesso (RBAC - Role Based Access Control)
O sistema terá dois níveis de permissão distintos:
ROLE_USER (Público): Pode apenas visualizar pets e enviar formulários de interesse. Não precisa de login.
ROLE_ADMIN (ONG): Requer login e senha para cadastrar, editar ou excluir pets, além de visualizar a lista de interessados.
10.2. Proteção da API (Spring Security)
Ambiente atual de desenvolvimento: rotas liberadas (permitAll) para facilitar integração.
Próxima etapa: adicionar autenticação JWT para rotas administrativas e armazenar senhas com BCrypt.
10.3. Privacidade de Dados (LGPD)
Minimização: Coletar apenas o necessário no formulário de interesse (Nome, E-mail, WhatsApp).
Consentimento: Adicionar um checkbox de "Aceito os termos de privacidade" no formulário de contato.

13. Ambiente de Banco de Dados (PostgreSQL)
13.1. Com Docker Compose
- Pré-requisito: Docker Desktop instalado e em execução
- Arquivo: docker-compose.yml na raiz do projeto
- Comando: docker compose up -d
- Serviço: db (postgres:16-alpine)
- Credenciais padrão:
  - Host: localhost
  - Porta: 5432
  - Database: tocadospeludos
  - User: postgres
  - Password: postgres

13.2. Sem Docker (instalação local)
- Criar o database tocadospeludos
- Aplicar o schema SQL:
  - psql -h localhost -U postgres -d tocadospeludos -f backend/src/main/resources/db/scripts/schema.sql

13.3. Variáveis de Ambiente do Backend
- DB_URL=jdbc:postgresql://localhost:5432/tocadospeludos
- DB_USERNAME=postgres
- DB_PASSWORD=postgres
- O Hibernate (spring.jpa.hibernate.ddl-auto=update) mantém o schema sincronizado com os Models.
